# Deploying Forge AI to Render

Forge AI runs as a **single Node web service** (the Next.js app, which also
hosts every API route, tRPC, BetterAuth, Inngest, and the GitHub/Razorpay
webhooks) backed by a **managed Postgres** database. Everything is wired by
[`render.yaml`](./render.yaml) — a Render Blueprint.

The app reads all of its data from the live database. There is **no demo/seed
data in the deploy path** — `packages/db/prisma/seed.ts` is an optional local
helper only and is never run on Render.

---

## 1. One-time deploy (Blueprint)

1. Push this repo to GitHub (the remote is already `origin`).
2. In Render → **New → Blueprint** → select the `forge-ai` repo.
3. Render reads `render.yaml` and provisions:
   - `forge-ai-db` — free managed Postgres (Singapore region).
   - `forge-ai-web` — the Node web service.
4. Click **Apply**. The first build runs:
   ```
   pnpm install
   prisma generate
   prisma db push        # creates the schema in the fresh DB
   next build
   ```
   then starts with `next start` (binds to Render's `$PORT`).

The service comes up at `https://forge-ai-web.onrender.com`. If your URL ends up
different (name collision), update `NEXT_PUBLIC_APP_URL` and `BETTER_AUTH_URL`
in the dashboard and redeploy.

---

## 2. Environment variables

`render.yaml` handles these for you:

| Variable | Source |
| --- | --- |
| `DATABASE_URL`, `DIRECT_URL` | auto-wired from `forge-ai-db` |
| `BETTER_AUTH_SECRET`, `ENCRYPTION_KEY`, `GITHUB_WEBHOOK_SECRET` | auto-generated by Render |
| `NEXT_PUBLIC_APP_URL`, `BETTER_AUTH_URL` | preset to the onrender.com URL |
| `NODE_ENV`, `NODE_VERSION` | preset |

Fill these in the **Render dashboard** (marked `sync: false`) to light up each
pipeline. The app boots and runs without them — features activate as keys are
added.

| Pipeline | Variables | Where to get them |
| --- | --- | --- |
| **GitHub sign-in + repo access** | `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` | github.com/settings/developers → New OAuth App. Callback: `https://forge-ai-web.onrender.com/api/auth/callback/github` |
| **AI (PRDs / reviews / tasks)** | `OPENAI_API_KEY` | platform.openai.com/api-keys |
| **Workflow engine** | `INNGEST_EVENT_KEY`, `INNGEST_SIGNING_KEY` | app.inngest.com → Settings → Keys (Production) |
| **Billing** | `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`, `RAZORPAY_WEBHOOK_SECRET`, `RAZORPAY_PLAN_PRO`, `RAZORPAY_PLAN_TEAM`, `NEXT_PUBLIC_RAZORPAY_KEY_ID` | dashboard.razorpay.com |

> Changing a `NEXT_PUBLIC_*` value requires a **redeploy** (it's inlined into the
> client bundle at build time).

---

## 3. Connecting the pipelines (after first deploy)

These steps register the live endpoints with the external services:

1. **Inngest** — in the Inngest dashboard, add your app and sync the endpoint
   `https://forge-ai-web.onrender.com/api/inngest`. Set `INNGEST_EVENT_KEY` /
   `INNGEST_SIGNING_KEY` in Render to the production keys.
2. **GitHub webhooks** — for each repo (or the GitHub App), point the webhook to
   `https://forge-ai-web.onrender.com/api/webhooks/github` and use the
   `GITHUB_WEBHOOK_SECRET` value Render generated.
3. **Razorpay webhook** — point it at
   `https://forge-ai-web.onrender.com/api/webhooks/razorpay` with
   `RAZORPAY_WEBHOOK_SECRET`.

---

## 4. Schema changes later

The blueprint runs `prisma db push` on every deploy, so editing
`packages/db/prisma/schema.prisma` and pushing to GitHub is enough to evolve the
production schema. (There are no migration files; the schema is the source of
truth. If you later want migration history, switch the build step to
`prisma migrate deploy` and start committing `prisma/migrations/`.)

---

## 5. Notes & limits

- **Free plan**: the web service spins down when idle (cold starts) and the free
  Postgres is deleted after ~30 days. Upgrade both to `starter` in `render.yaml`
  for always-on production use.
- **Region** is set to `singapore` (closest to India). Change `region:` in
  `render.yaml` if you prefer another.
- The optional demo data (`pnpm db:seed`) is for local exploration only — do not
  run it against the production database.
